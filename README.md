# SDLC Agent

**SDLC Agent** — инструмент для автоматизации цикла разработки программного обеспечения (Software Development Lifecycle) с использованием больших языковых моделей.

Система автоматически:

1. Анализирует **GitHub Issue**
2. Генерирует **код**
3. Создаёт **Pull Request**
4. Запускает **CI**
5. Выполняет **AI-ревью кода**
6. При необходимости **исправляет код и запускает новую итерацию**

Цикл повторяется до успешного ревью или достижения лимита итераций.

---

# Обзор

Современные LLM способны генерировать код, однако их интеграция в реальные процессы разработки остаётся сложной задачей.

Этот проект реализует **автоматизированный цикл разработки**, объединяющий:

- GitHub Issues
- Pull Requests
- CI pipeline
- генерацию кода с помощью LLM
- автоматическое AI-ревью

Цель проекта — создать **самоисправляющийся pipeline разработки**, способный реализовывать задачи из Issue.

---

# Архитектура системы

Система состоит из двух основных компонентов.

## Code Agent

**Code Agent** — LLM-управляемый агент разработки, который реализует задачи из GitHub Issue.

Он выполняет следующие действия:

- анализирует описание задачи
- генерирует код
- создаёт ветку
- создаёт Pull Request
- исправляет код после ревью

---

## AI Review Module

**AI Reviewer** — модуль проверки, основанный на LLM.

Он анализирует:

- diff Pull Request
- описание Issue
- результаты CI

После анализа модуль возвращает структурированный результат:

```
status=approved
```

или

```
status=needs-fix
```

На основе этого результата система либо завершает цикл, либо запускает новую итерацию Code Agent.

---

# Цикл работы системы

```
Создание Issue
      │
      ▼
 Code Agent
 (генерация кода)
      │
      ▼
 Pull Request
      │
      ▼
 CI (black + pytest)
      │
      ▼
 AI Reviewer
      │
      ├── approved → завершение
      │
      └── needs-fix
             │
             ▼
         Code Agent
        (новая итерация)
```

# Пример конфигурации LLM

Пример для HuggingFace Router:

```json
{
  "mode": "openai_chat",
  "base_url": "https://router.huggingface.co",
  "api_key": "hf_xxx",
  "model": "openai/gpt-oss-120b:fastest",
  "timeout_s": 60,
  "temperature": 0.2,
  "max_tokens": 3000,
  "headers": {
    "Authorization": "Bearer ${api_key}"
  },
  "endpoints": {
    "chat_completions": "/v1/chat/completions"
  }
}
```

## Описание параметров

| Параметр | Тип | Описание |
|---------|-----|----------|
| `mode` | string | Режим подключения к модели. Поддерживаются: `openai_chat`, `ollama_chat`, `hf_inference`, `mock`. |
| `base_url` | string | Базовый URL сервера инференса. Может указывать на HuggingFace Router, локальный сервер vLLM, Ollama, llama.cpp или любой OpenAI-совместимый API. |
| `api_key` | string | API ключ для доступа к модели. Для локальных моделей может быть пустым. |
| `model` | string | Название модели, используемой для инференса. Например: `openai/gpt-oss-120b:fastest`, `deepseek-coder`, `llama3`. |
| `timeout_s` | integer | Максимальное время ожидания ответа от модели (в секундах). Если ответ не получен за это время, запрос считается неуспешным. |
| `temperature` | float | Параметр случайности генерации. Меньшие значения делают ответы более детерминированными, большие — более разнообразными. |
| `max_tokens` | integer | Максимальное количество токенов в ответе модели. Ограничивает размер генерируемого текста. |
| `headers` | object | Дополнительные HTTP-заголовки для запросов к API. Поддерживается подстановка переменных, например `${api_key}`. |
| `endpoints` | object | Переопределение путей API endpoint. Используется, если сервер использует нестандартные пути для запросов. |

## Конфигурация SDLC

Поведение агента можно настроить через файл:

```
.sdlc-agent/config.json
```

Пример конфигурации:

```json
{
  "default_branch": "main",
  "max_iterations": 3,
  "review": {
    "require_ci_success": true
  }
}
```

### Параметры

| Параметр | Описание |
|--------|----------|
| `default_branch` | Базовая ветка, относительно которой создаются Pull Request |
| `max_iterations` | Максимальное количество итераций исправления кода |
| `review.require_ci_success` | Требовать ли успешного прохождения CI перед одобрением PR |

# Использование

После установки workflow система работает автоматически.

### 1. Создать Issue

Например:

```
Add function reverse_string in src/reverse.py
```

---

### 2. Code Agent создаёт Pull Request

```
Implement #1: reverse_string
```

---

### 3. CI выполняет проверки

- black
- pytest

---

### 4. AI Reviewer анализирует PR

Если обнаружены ошибки:

```
status=needs-fix
```

Code Agent автоматически создаёт новую итерацию.

## Ограничения

Текущая версия системы имеет ряд ограничений:

- ориентирована на **Python-проекты**
- CI предполагает наличие инструментов **`black` и `pytest`**
- корректность генерации зависит от качества **описания GitHub Issue**
- система анализирует **только diff Pull Request**, а не весь проект целиком
- AI Reviewer выполняет семантическую проверку и не гарантирует алгоритмическую корректность решения
- качество генерации зависит от используемой **LLM модели**

## Дальнейшее развитие

Несмотря на работоспособность текущей версии системы, существует ряд направлений для её дальнейшего развития и улучшения.

### Улучшение этапа планирования (Planning)

В текущей версии системы этап планирования используется в упрощённом виде.  
Модель формирует список шагов реализации задачи, который затем используется Code Agent при генерации кода.

Возможные улучшения:

- генерация более структурированного **плана реализации**
- автоматическое формирование **acceptance criteria**
- декомпозиция задачи на **подзадачи**
- использование плана для более точного управления генерацией кода

Это позволит уменьшить количество ошибок генерации и повысить стабильность работы системы при сложных задачах.

---

### Более сложные стратегии ревью

Текущий AI Reviewer выполняет анализ изменений на основе:

- diff Pull Request
- описания задачи (Issue)
- результатов CI

В дальнейшем возможно расширение логики ревью:

- анализ архитектурных изменений
- проверка соответствия стилю проекта
- выявление потенциальных ошибок
- анализ edge-cases

Также возможна реализация **многоэтапного ревью**, где разные модели выполняют различные типы анализа.

---

### Поддержка multi-file reasoning

В текущей реализации модель анализирует изменения, но не всегда учитывает полный контекст проекта.

В будущем возможно:

- загрузка нескольких связанных файлов в контекст модели
- анализ зависимостей между модулями
- улучшение reasoning модели при работе с крупными изменениями

Это позволит системе корректно обрабатывать задачи, затрагивающие несколько частей проекта.

---

### Улучшение метрик качества

В текущей версии система демонстрирует работоспособность, однако оценка качества может быть расширена.

Возможные метрики:

- процент успешно созданных Pull Request
- количество итераций исправления до успешного ревью
- доля задач, принятых без дополнительного вмешательства
- среднее время выполнения полного цикла задачи

Также возможно проведение сравнительного анализа различных моделей для оценки их эффективности.

---

### Расширение поддерживаемых языков

Текущая версия системы ориентирована на Python-проекты.

В будущем возможно добавление поддержки других языков:

- JavaScript / TypeScript
- Java
- Go
- C++

Это потребует адаптации CI pipeline и стратегий анализа кода.

---

### Архитектурное развитие системы

Дальнейшее развитие системы может включать переход к более сложной агентной архитектуре:

- использование **LangGraph** для управления workflow
